Dev Take

Went with discriminated unions over STI — nullable columns everywhere felt wrong. The audit log instead of soft deletes was the right call too.

Execution Path

Schema design
Sketched out the core models — User, Account, Transaction. Decided on a discriminated union for transaction types instead of STI.
Prisma setup
Initialized Prisma with PostgreSQL, wrote the schema file with proper relations and indexes.
Migration
Ran the initial migration, fixed a cascade delete issue on Account → Transaction.
Seed data
Created a seed script with realistic test data — multiple accounts, mixed transaction types.

Q&A

  • Why did you choose a discriminated union over single-table inheritance for transaction types?
    STI would have left nullable columns everywhere — transferTo only makes sense for transfers, recurringSchedule only for recurring. The union pattern keeps each type clean and the TypeScript compiler catches missing fields.
  • How does the cascade delete work between Account and Transaction?
    Deleting an account cascades to its transactions. Initially had it set to restrict, but that made account cleanup impossible without manual transaction deletion first.
