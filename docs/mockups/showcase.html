Started BudgetWise by designing the Prisma schema around discriminated unions for transaction types -- checking, savings, credit, investment accounts each get their own shape instead of nullable fields everywhere. This paid off immediately when building the transaction list: TypeScript catches invalid field access at compile time, and the infinite-scroll query stays simple because there is no conditional logic per account type.

The recurring transaction detection was the hardest optimization. The naive approach scanned every transaction pair looking for periodic patterns, which took 2 seconds on a modest dataset. Rewrote it with a partial index on (merchant, amount, account_id) and a window function that checks day-of-month variance. Detection dropped to 120ms, and the partial index means the write path barely notices.

Plaid webhook integration taught me that idempotency is not optional for financial data. Each webhook carries a unique item_update_id, and the handler checks a processed_webhooks table before touching any transaction records. Added a dead-letter queue for failed webhook processing -- three retries with exponential backoff, then surface it in the admin dashboard for manual review.
